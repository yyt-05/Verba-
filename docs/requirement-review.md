# AI 同声传译助手 - 多角色需求评审

评审时间：2026-06-05 17:44

评审方式：多智能体角色讨论

## 参与角色

| 角色 | 评审重点 |
|---|---|
| 产品经理（PRD 撰写人） | 需求背景、目标用户、核心功能、主流程、边界规则、MVP 范围 |
| 前端开发 | 页面实现、悬浮窗、Flutter 客户端、SSE 状态管理、音频采集成本 |
| 后端开发 | 数据库、接口、ASR/翻译/修正流水线、SSE、性能与成本风险 |

## 一句话结论

需求方向成立，核心价值是：

> 不依赖原视频字幕，通过桌面系统音频捕获 + ASR + 实时翻译 + 悬浮双语字幕 + 后验修正，帮助用户看懂英文视频、网课、会议录播。

但当前需求范围偏大，尤其是“三端覆盖 + 系统音频捕获 + 实时 ASR + 翻译 + 修正 + SSE + 悬浮窗”同时进入 MVP，会显著增加工程风险。多角色一致建议：第一版 MVP 收敛为 **Windows 桌面端视频实时双语字幕工具**。

## 核心共识

### 1. MVP 应先聚焦 Windows 桌面端

建议第一版 MVP 聚焦：

- Windows 桌面端
- 系统音频捕获（WASAPI loopback）
- 英文 ASR
- 中文翻译
- 悬浮窗双语字幕
- 基础后验修正
- 修正高亮
- 一键开始/停止
- 断线、静音、权限、API 错误等基础异常处理

暂缓：

- Web 端完整能力
- iOS
- Android 系统音频捕获
- macOS 系统音频捕获
- 语音播报/TTS
- 术语库
- 用户登录
- 历史记录云同步
- 商业化/付费体系
- 多语种完整覆盖

### 2. 产品定位需要更准确

当前文档使用“同声传译助手”，但实际 MVP 更接近：

- AI 实时字幕翻译助手
- AI 双语字幕助手
- 实时视频字幕翻译工具

严格意义上的“同声传译”通常包含实时语音输出，而当前 MVP 主体是“实时字幕 + 后验修正”。如果继续使用“同声传译”，需要在 PRD 中明确 MVP 仅做字幕，不做语音播报。

### 3. 后验修正是差异化，但必须克制

自动修正是竞赛创新点，但不能无限回改历史字幕。建议规则：

- 每条字幕必须有稳定 `segmentId`
- 修正事件必须按 `segmentId` 回写，不能按文本匹配
- 修正只允许覆盖最近 6-8 个 segment
- 已展示超过 30 秒的字幕默认不再修正
- 修正事件必须带 `revision`
- 客户端只接受更高版本的 revision
- UI 只高亮被修正句，不整窗闪烁
- 高亮时长控制在约 500ms

### 4. 客户端最大风险不是页面，而是音频与悬浮窗

Flutter 可以承担主界面、字幕 UI、状态管理和 SSE 接收，但 Windows 系统音频捕获大概率需要 Native 插件封装。

客户端风险优先级：

| 风险 | 等级 | 处理建议 |
|---|---:|---|
| Windows 系统音频捕获封装不稳定 | 高 | 先做技术 Spike，保留麦克风兜底 |
| 悬浮窗置顶、拖拽、多显示器适配 | 中高 | MVP 先做置顶、拖拽、缩放，不强求点击穿透 |
| correction 回写错位 | 中高 | 客户端维护 `segmentId` 与 `revision` |
| 异常 UI 不完整 | 中 | P0 覆盖权限、断线、静音、API 错误 |

### 5. 后端必须显式建模会话和字幕片段

后端不能做成“传一段音频，返回一段翻译”的同步接口。实时场景需要显式建模：

- session
- audio chunk
- subtitle segment
- correction event
- SSE event cursor
- usage/cost event

推荐核心链路：

```text
音频采集 → 音频上传 → VAD/分片聚合 → ASR → 翻译 → 修正 → SSE 推送 → 客户端渲染
```

推荐内部队列：

```text
audio_chunk_queue
  → asr_result_queue
  → translation_queue
  → correction_queue
  → sse_event_queue
```

## 角色评审结论

### 产品经理（PRD 撰写人）

需求背景成立，但当前 PRD 偏“方案先行”，还需要补充用户问题证据。

建议补充 2-3 个用户故事：

- 技术从业者观看无字幕英文发布会
- 学生观看英文网课
- 用户参加英文线上会议或技术分享

PRD 需要进一步明确：

- MVP 优先用户：桌面端观看英文视频/网课/会议录播的用户
- 手机端场景：身边人说话、现场演讲，不应承诺捕获 YouTube/Zoom 内部音频
- 成功标准：3-5 秒内出现中文字幕，连续 10 分钟不中断，字幕窗口可拖动调整
- 边界声明：不支持多人分离，不保证专业术语完全准确，不做会议记录归档，不做语音播报

### 前端开发

MVP 可做，但客户端复杂度主要集中在：

- Windows 系统音频捕获
- 悬浮窗能力
- SSE 事件状态机
- 字幕 store
- correction 回写
- 异常状态 UI

建议页面结构：

- 首页/悬浮控制按钮
- 悬浮字幕窗
- 状态与异常提示层
- 简化设置页

客户端建议状态枚举：

```text
idle
requestingPermission
capturing
connecting
listening
silent
reconnecting
apiError
permissionDenied
audioSourceUnavailable
stopped
```

工期估算：

| 范围 | 估算 |
|---|---:|
| 完整客户端 MVP | 25-38 人日，约 4-6 周 |
| Windows 单端竞赛 Demo | 15-22 人日，约 3-4 周 |

### 后端开发

MVP 可以成立，但需要从一开始设计会话、分片、事件、成本控制，否则后续会在延迟、重连、修正和费用上失控。

推荐接口：

```http
POST /api/v1/sessions
POST /api/v1/sessions/{sessionId}/audio
GET  /api/v1/sessions/{sessionId}/events
POST /api/v1/sessions/{sessionId}/stop
```

SSE 事件类型建议：

```text
session.status
subtitle.partial
subtitle.final
subtitle.corrected
warning
error
usage
```

关键后端规则：

- `chunkSeq` 必须幂等
- 音频格式建议统一为 `16kHz mono pcm16` 或 `opus/webm`
- 服务端做 VAD，静音片段不要送 ASR
- SSE 事件必须有递增 `id`
- 客户端重连时带 `Last-Event-ID`
- 服务端保留最近 5 分钟或最近 500 条事件
- 每个 session 设置预算上限
- 单用户最多 1 个 active session
- 单会话最长 30-60 分钟

### UI/UX 设计师

设计重点不是页面数量，而是悬浮窗长期覆盖体验、修正反馈尺度和异常状态闭环。

P0 页面/模块：

- 悬浮窗入口按钮
- 字幕悬浮窗
- 权限引导弹窗
- 连接/重连状态层
- 无语音提示
- API Key/额度错误页

悬浮窗建议：

- 桌面端默认底部居中
- 宽度约屏幕 60%-80%
- 半透明深色背景
- 中文为主视觉，英文为辅助视觉
- 支持拖拽、折叠、透明度、字号、停止/继续
- 用户手动回看时暂停自动滚动到底部
- 提供“回到底部”按钮

设计交付物建议：

- 核心体验流程图
- 页面/状态矩阵
- 桌面端悬浮窗高保真稿
- 移动端麦克风拾音稿
- 修正高亮动效说明
- 异常状态文案表

### 测试工程师（QA）

测试重点覆盖 MVP 主链路、异常恢复、性能、兼容性、安全和成本边界。

P0 测试范围：

- 一键开始/停止
- Windows 系统音频捕获
- 麦克风录音兜底
- ASR 识别
- 翻译
- 双语字幕展示
- 修正引擎
- 修正高亮动画
- SSE 推送与重连
- 权限拒绝
- 静音 30 秒提示
- API 超时、Key 无效、额度耗尽

验收标准建议：

- 用户打开 App 后，可一键开始监听
- Windows 桌面端可捕获系统音频并实时生成双语字幕
- 英文原文和中文翻译能按顺序展示
- 端到端延迟稳定控制在 2-4 秒内，或 Demo 版本明确放宽到 3-5 秒
- SSE 可推送普通字幕和修正事件
- 修正事件能更新已有字幕并触发约 500ms 黄色高亮
- 停止监听后不再上传音频、不再调用 API，字幕保留可回看
- 连续运行 1 小时无崩溃、无明显内存泄漏、无严重乱序

## 主要需求漏洞

| 问题 | 影响 | 建议 |
|---|---|---|
| “字幕或语音形式呈现”范围不清 | 是否需要 TTS 会影响工期和成本 | MVP 明确只做字幕 |
| 多语种范围不清 | 影响 ASR、翻译、测试范围 | MVP 明确英语到中文 |
| 三端覆盖与 Web 能力冲突 | Web 无法捕获系统音频 | Web 暂缓或只做展示端 |
| macOS 是否纳入 MVP 不清 | 影响音频捕获实现 | MVP 先不验收 macOS |
| 修正准确率标准缺失 | 难以验收核心创新点 | 建立样例集和人工评测口径 |
| 延迟统计口径不清 | 难以判断是否达标 | 定义为“音频输入时间戳 → 中文字幕首次显示” |
| 字幕分句规则未定义 | 影响可读性、延迟、修正定位 | 统一使用 segment |
| 停止后回看与历史记录边界不清 | 影响存储和隐私 | MVP 只保留当前会话内存态 |
| 隐私合规缺失 | 采集会议/麦克风内容有风险 | 明确音频是否落盘、字幕是否保存、日志策略 |
| 成本控制未产品化 | API 费用可能失控 | 加单会话时长、并发、静音、重试、预算上限 |

## 建议落地顺序

### 第 0 阶段：技术 Spike

目标：先验证最大技术风险。

- Windows WASAPI loopback 音频捕获
- Flutter 悬浮窗置顶、拖拽、缩放
- SSE Demo 接收 subtitle/correction
- 后端音频分片上传 Demo

验收：

- 能捕获正在播放的视频声音
- 能在悬浮窗显示服务端推送的双语字幕
- 能按 `segmentId` 修正一条历史字幕

### 第 1 阶段：MVP 主链路

目标：打通最小完整闭环。

- 一键开始/停止
- 音频采集和上传
- ASR
- 翻译
- SSE 推送
- 双语字幕展示
- 停止后保留当前字幕

验收：

- 播放英文视频后，3-5 秒内出现中文字幕
- 连续 10 分钟字幕不中断
- 停止后不再上传音频和调用 API

### 第 2 阶段：修正能力

目标：实现差异化能力。

- 上下文滑动窗口
- correction 事件
- revision 版本控制
- UI 高亮修正
- 限制修正范围和频率

验收：

- 修正只更新指定字幕条目
- 不出现重复字幕
- 高亮不造成整窗闪烁或滚动错位

### 第 3 阶段：异常与演示稳定性

目标：让 Demo 不容易翻车。

- 网络断线自动重连
- 静音 30 秒提示
- API 失败重试 3 次
- API Key/额度启动校验
- 权限拒绝引导
- 音频源不可用时切换麦克风兜底

验收：

- 断网、静音、权限拒绝、API 错误均有明确状态
- 连续运行 1 小时不崩溃

### 第 4 阶段：体验打磨

目标：提升观感和可用性。

- 字幕样式设置
- 透明度设置
- 字号设置
- 折叠/展开
- 手动回看暂停自动滚动
- 回到底部按钮
- Android 麦克风版

## 推荐 MVP 范围定义

### MVP 包含

- Windows 桌面端
- 系统音频捕获
- 麦克风兜底
- 英文到中文
- 实时双语字幕
- 一键开始/停止
- SSE 实时推送
- 基础后验修正
- 修正高亮
- 权限、断线、静音、API 错误处理

### MVP 不包含

- TTS 语音播报
- 多语种完整支持
- Web 系统音频捕获
- iOS
- macOS 系统音频捕获
- Android 捕获其他 App 系统音频
- 用户登录
- 云端历史记录
- 术语库
- 商业化付费
- 多人说话人分离

## 后续 PRD 需要补充的问题

1. MVP 是否正式收敛为 Windows 桌面端？
2. MVP 是否只支持英语到中文？
3. 是否确认不做 TTS 语音播报？
4. Web 端是完整客户端、展示端，还是暂缓？
5. macOS 是否纳入第一版验收？
6. 原始音频是否完全不落盘？
7. ASR 文本和翻译文本是否允许保存用于调试？
8. 字幕停止后是否只保留当前会话内存态？
9. 修正允许回改多早之前的字幕？
10. 修正成功率、误修率如何验收？
11. 端到端延迟以什么时间点为起止？
12. 单会话最长使用时长是多少？
13. 是否需要每日额度或单用户预算上限？
14. API Key 是服务端统一持有，还是用户自带？
15. 竞赛 Demo 目标并发是多少？

## 最终建议

第一版不要做“三端完整 AI 同传产品”，而是做一个足够稳定、足够聚焦的：

> Windows 桌面 AI 实时双语字幕助手

以“自动监听 YouTube/网课/会议录播音频并生成悬浮中文字幕”为主线，把系统音频捕获和低延迟字幕体验做扎实；后验修正作为核心创新点保留，但先做克制版本。这样最贴合原始需求中“悬浮窗按钮 → 点一下开启 → 自动监听 YouTube 视频里的外语 → 实时显示弹幕式中英文字幕”的关键体验，也能显著降低 MVP 落地风险。
