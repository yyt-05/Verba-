# AI 同声传译助手

用户经常需要观看英语（或其他外语）演讲、技术分享、国际会议或网课，请开发一款AI同声传译助手，帮助用户降低语言门槛，提升信息获取效率。
要求：通过 AI 能力，将单向音频流实时、流畅地翻译成中文，以字幕或语音形式呈现，帮助用户跟上内容节奏。系统需具备修正能力，能够自动纠正之前识别或翻译的错误。


**一个悬浮窗按钮 → 点一下开启 → 自动监听YouTube视频里的外语 → 实时显示"弹幕式"的中英文字幕**


## 一、用户需求分析

> 核心理念：任何英文语音进来，都能实时翻译成中文。

### 1. 目标用户定位

**核心用户画像**：

| 维度 | 描述 |
|------|------|
| 年龄 | 22-40 岁 |
| 职业 | 互联网/科技从业者、高校学生、研究者、产品经理 |
| 特征 | 有阅读英文文档能力，但**听力不足以跟上英文原速** |
| 痛点 | 听不懂英文内容，无法参与英文会议/活动，被语言门槛限制信息获取 |

**使用场景**：

| 场景 | 音频来源 | 收音方式 | 典型情况 |
|------|---------|---------|---------|
| **场景1：看视频** | YouTube/网课/会议录播 | 系统音频捕获（桌面端）/ 麦克风拾音（手机端） | 在家或咖啡厅刷英文技术分享 |
| **场景2：身边人说英文** | 周围人在讲英文 | 手机麦克风 | 同事在旁边英文开会、外国朋友聊天、参加线下英文 meetup |
| **场景3：现场演讲/会议** | 讲台上的演讲者 | 手机麦克风（远场拾音） | 参加国际会议、行业大会、英文培训 |
| **场景4：线上会议** | Zoom/Teams/腾讯会议 | 系统音频或麦克风 | 参加全英文线上会议，听不懂但需要跟上 |

MVP 阶段聚焦场景1 + 场景2（技术上是同一套能力：音频进入 → 识别 → 翻译 → 显示，仅音频来源不同）。

**MVP 不区分用户层级**：不设普通/付费/高频用户划分，所有用户使用同一套能力。

### 2. 需求本质拆解

```
显性需求：「我想听懂英文内容，给我中文翻译」
                    ↓
隐性需求：「翻译错了要告诉我，别让我被错误翻译带偏」
                    ↓
深层需求：「我需要信任这个翻译结果，才能放心使用」
```

**KANO 模型优先级**：

| 类型 | 功能 | 说明 |
|------|------|------|
| **必备型** | 语音识别 + 翻译 + 字幕显示 | 没有这些，产品不成立 |
| **期望型** | 低延迟、翻译流畅自然 | 做得越好用户越满意，做得差用户会走 |
| **魅力型** | 自动修正 + 修正高亮闪烁 | 用户不会主动要求，但有了会超出预期 — **竞赛创新性核心得分点** |
| 无差异型 | 用户登录、历史记录 | MVP 可有可无 |

### 3. 用户体验评估

| 维度 | 目标 |
|------|------|
| **易用性** | 打开 App → 点一下按钮 → 开始看翻译。零学习成本，无需复杂配置 |
| **可用性** | 延迟控制在 2-4 秒以内（参考豆包 2-3 秒）；字幕流畅滚动，不卡顿 |
| **情感反馈** | 修正高亮是最关键的「信任感」设计 — 让用户知道系统在主动纠错，而非默默修改 |

## 二、市场与竞品分析

### 1. 市场环境洞察

- **行业趋势**：AI 语音翻译赛道正处于快速爆发期。2025 年字节跳动发布 Seed LiveInterpret 2.0 同传模型（2-3 秒延迟），豆包同声传译以免费策略进入市场；讯飞听见持续迭代 AI 转写能力；Google、Meta 等巨头持续投入多语种翻译模型。生成式 AI（GPT-4o、Whisper）大幅降低了语音识别和翻译的门槛，使得小团队也能构建高质量的同传产品。
- **竞争格局**：头部玩家（豆包/字节、讯飞、有道）占据品牌和流量优势，但产品形态偏「工具型」—— 要么是语音转文字工具，要么是对话翻译工具。腰部产品（沉浸式翻译、Shinkansen 等浏览器插件）专注 YouTube 字幕翻译细分场景，用户量大但依赖已有 CC 字幕。**目前没有产品将「实时语音识别 + 翻译 + 自动修正」整合成一个面向视频观看场景的完整体验。**

### 2. 竞品对比分析

本产品横跨两个赛道，竞品也分为两类：

#### 赛道一：浏览器插件（YouTube 字幕翻译）

| 产品 | 用户量 | 双语字幕 | AI 引擎 | 核心模式 | 致命弱点 |
|------|--------|:--:|--------|----------|----------|
| **沉浸式翻译** | 1000万+ | ✅ | 多 AI | 网页+视频字幕翻译 | 依赖原视频 CC 字幕，无字幕视频无法工作 |
| **Shinkansen** | 开源 | ✅ | Gemini/GPT | 极速字幕翻译 | 同上，依赖已有字幕 |
| **YouTube 双语字幕翻译助手** | Chrome 插件 | ✅ | GPT-4/Gemini 等 | 双语字幕对照 | 同上 |
| **PolyDub** | Firefox 插件 | ✅ | Azure/Google TTS | 字幕翻译+AI 配音 | 依赖 CC 字幕 |

**共同致命缺陷**：它们都依赖 YouTube **已有的 CC 字幕**做翻译，本质是「字幕翻译器」而非「语音翻译器」。大量技术分享、会议录播没有字幕，这类工具完全无法工作。

#### 赛道二：独立 App / SDK（语音 → 翻译）

| 产品 | 语音→文字 | 实时翻译 | 延迟 | 价格 | 关键弱项 |
|------|:--:|:--:|------|------|------|
| **豆包同声传译** | ✅ Whisper 级 | 中↔英等 7 语 | 2-3 秒 | 免费 | 无修正能力；产品形态为「语音→语音」，无字幕模式；无桌面悬浮窗 |
| **讯飞听见** | ✅ 98% 准确率 | 8-10 语 | 实时 | ¥18-88/月 | 偏「录音转写」场景，非实时同传；无翻译修正；无系统音频捕获 |
| **有道翻译官** | ✅ | 多语种 | 实时 | 免费+付费 | 偏「对话翻译」场景；无后验修正 |
| **Google Live Transcribe** | ✅ 优秀 | 100+ 语 | 实时 | 免费 | 仅 Android；无翻译修正；中国大陆不可用 |
| **Microsoft Translator** | ✅ | 100+ 语 | 实时 | 免费 | 偏「对话模式」；无悬浮窗；无修正 |

### 3. 竞品定位图

```
        依赖已有字幕 ←──────────────────→ 自己识别语音
              │                                    │
 沉浸式翻译    │                   本产品 ★          │
 Shinkansen   │                   (ASR + 翻译       │
 PolyDub      │                    + 修正引擎)       │
              │                                    │
 ───────────────────────────────────────────────────
              │                                    │
    讯飞听见  │                        豆包同传     │
  (录音转写)  │                      (语音→语音)    │
              │                                    │
       偏「事后」                         偏「实时对话」
```

本产品位于右上角 — **自己识别语音 + 实时翻译 + 自动修正**，目前这个位置没有直接竞品。

### 4. 核心差异化总结

| 差异点 | 本产品 | 所有竞品 |
|--------|:------:|:------:|
| **后验自动修正引擎** | ✅ 上下文滑动窗口 + LLM 回溯纠正 | ❌ 无一具备 |
| **系统音频捕获** | ✅ 桌面端 WASAPI loopback | ❌ 均依赖麦克风或 CC 字幕 |
| **悬浮窗双语字幕** | ✅ 弹幕式悬浮窗覆盖在视频上 | ❌ 豆包无字幕模式，插件仅翻译已有字幕 |
| **不依赖原视频字幕** | ✅ 自建 ASR 识别语音 | ❌ 插件类全部依赖 CC 字幕 |
| **三端覆盖** | ✅ Android + Windows + Web | ❌ 大多是单端产品 |

**核心结论**：竞品验证了「修正引擎 + 悬浮窗 + 系统音频捕获」是真实的市场空白。豆包、讯飞、有道三家国内头部，无一具备翻译后验修正能力，这是竞赛「创新性」评分的核心得分点。

### 2. 音频采集策略：按平台自动决定

**各平台系统音频捕获可实现性**：

| 平台 | 系统音频捕获 | 原因 |
|------|:--:|------|
| Windows 桌面 | ✅ 可以 | WASAPI loopback，无限制 |
| macOS 桌面 | ✅ 可以 | ScreenCaptureKit / BlackHole |
| Android 手机 | ❌ 不行 | AudioPlaybackCapture API 需要被录 App 声明同意，YouTube/浏览器/Zoom 全都没开 |
| iOS 手机 | ❌ 不行 | 系统未公开此 API |
| Web | ❌ 不行 | 浏览器沙箱不允许捕获系统音频 |

**结论**：桌面端用系统音频，手机端用麦克风。用户不需要手动选择，程序按平台自动走不同路径：

```
打开 App → 点「开始监听」
              │
              ▼
     ┌── 桌面端（Windows/macOS）？
     │   → 系统音频捕获（WASAPI / ScreenCaptureKit）
     │   → 覆盖场景：看视频、线上会议（戴耳机也能捕获）
     │
     └── 手机端（Android/iOS）？
         → 麦克风录音
         → 覆盖场景：身边人说话、现场演讲
```

**异常场景处理**：

| 异常 | 处理方式 |
|------|---------|
| 网络中断 | SSE 连接断开 → 自动重连 → 期间显示「连接中...」 |
| ASR/翻译 API 失败 | 重试 3 次 → 仍失败跳过当前句 → 继续处理下一句 |
| 长期无语音输入 | 静音超过 30 秒 → 提示「未检测到语音，请检查音频源」 |
| 音频权限被拒绝 | 弹窗引导用户去系统设置开启权限 |
| API Key 无效/额度耗尽 | 启动时校验 → 不可用直接提示，不进入主界面 |
